会议纪要
日期：2026年3月4日
时间：下午 18:30 - 20:00
地点：图书馆N座

1. 会议目的
本次会议的主要目的是：
讨论并确认失败条件与风险管理机制的最终方案；
设计游戏失败结局的触发逻辑；
整理并汇报当前的工作进度与遇到的难点。
当前工作分工布置

2. 会议内容

2.1 失败条件与风险讨论

强调了失败条件的重要性，确保游戏有明确的失败节点以增强玩家体验。
确定了资金、声誉和凝聚力作为主要的失败触发条件。当这些资源下降到零时，系统会自动触发游戏失败。
讨论了失败条件应与剧情紧密结合，如资金告急会触发“破产”的剧情，影响后续任务。
确定了需要设计不同的失败结局界面，以便根据不同的失败条件（如资金或声誉）进行区分。

2.2 失败结局的触发与展示

讨论了多个失败结局的设计，包括：

资金告急结局：玩家未能及时做出正确决策，导致资金为零，触发酒吧倒闭的结局，后续剧情可能转向“重建”或“退出”方向。
声誉崩溃结局：声誉过低时触发员工不满、社交圈断裂的剧情，最终导致乐队解散，酒吧倒闭。
讨论了如何在游戏中实时跟踪资源的变化，并在合适的时机触发失败结局。
设计了失败界面，目前构思主要使用简洁的弹窗形式呈现失败信息，并通过动态效果展示资源低下的状况。

2.3 工作进度更新与问题讨论

整体项目进度报告：

剧情策划：已完成大部分主线和支线剧情的草案，部分分支剧情仍在完善中。
程序开发：资源管理系统的初步搭建已完成，接下来将完善事件触发机制。
UI设计：前期UI框架已完成，但仍需根据剧情推进进行调整。
美术与测试：主场景的初步素材已完成，测试团队正在进行初步功能测试。

2.4 风险讨论与应对措施

讨论了项目开发中可能出现的风险，特别是在技术实现与设计需求之间的差距。建议加强沟通，确保技术可行性与设计需求的对接。
对于复杂剧情分支，提出了技术实现的挑战，建议剧情策划团队与程序开发团队保持紧密合作，确保分支的可实现性。
针对UI设计，强调了动态效果与资源管理交互可能对性能的影响，建议提前进行性能测试。
美术团队提出，部分视觉效果在低端设备上可能会存在性能问题，建议进行更多优化。

3. 会议结论与行动项

3.1 确定事项

确定了失败条件与结局触发的逻辑，并对不同的失败结局进行了视觉设计。
确定了每个失败结局的触发条件，并设计了相应的失败信息界面。

3.2 下一步工作

完善分支剧情，尤其是与失败条件相关的内容。
完成失败结局触发的代码实现，确保系统在资源下降到一定值时能够触发相关结局。
优化失败结局的UI界面，并加强视觉效果。
继续测试失败结局界面的流畅度，进行视觉表现的优化。
