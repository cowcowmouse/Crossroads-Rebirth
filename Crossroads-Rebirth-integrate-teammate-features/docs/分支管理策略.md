# Git分支管理策略

## 🌿 分支结构

main (生产分支)
↑
release/* (发布分支)
↑
develop (开发主分支)
↑
feature/* (功能分支)
fix/* (修复分支)

## 📋 分支说明

### main 分支
- 保护分支，**禁止直接推送**
- 只接受 release 分支的合并
- 每个提交对应一个生产版本

### develop 分支
- 开发主分支，**禁止直接推送**
- 所有 feature/fix 分支从这里创建
- 合并到这里后自动触发测试

### feature/* 分支
- 命名：`feature/功能名称` (如 `feature/action-point-system`)
- 从 develop 创建
- 完成后合并回 develop

### fix/* 分支
- 命名：`fix/bug描述` (如 `fix/ui-display-error`)
- 从 develop 创建
- 修复后合并回 develop

### release/* 分支
- 命名：`release/v版本号` (如 `release/v0.1.0`)
- 从 develop 创建
- 测试通过后合并到 main 和 develop

## 🔄 工作流程

### 开发新功能
```bash
git checkout develop
git pull
git checkout -b feature/your-feature
# 开发...
git add .
git commit -m "feat: 完成某功能"
git push origin feature/your-feature
# 在GitHub创建PR到develop

合并流程
创建Pull Request

通过CI测试

至少1人Code Review通过

合并到目标分支

🚫 禁止操作
❌ 直接推送 main/develop

❌ 强制推送 (force push)

❌ 合并未经Review的代码
